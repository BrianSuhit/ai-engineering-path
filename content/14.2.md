A nivel de arquitectura de sistemas, integrar un LLM sin protección implica un flujo de datos directo y peligroso: el usuario envía un prompt, viaja a la API del modelo, y la respuesta vuelve cruda a la pantalla del usuario (User -&gt; LLM -&gt; User). Cuando decidimos implementar un sistema de AI Guardrails, lo que hacemos es alterar este flujo inyectando interceptores lógicos. El nuevo flujo seguro se transforma en: User -&gt; [Input Guardrail] -&gt; LLM -&gt; [Output Guardrail] -&gt; User.
<br><br>
La primera frontera de protección es el Input Guardrail (o Input Guard). Esta es una función de código que se ejecuta en tu servidor antes de realizar cualquier petición HTTP al proveedor del modelo (como Google u OpenAI). Su objetivo principal es proteger al sistema de ataques y evitar la fuga de datos sensibles. Por ejemplo, acá es donde interceptamos el texto para aplicar PII Redaction (ocultar números de teléfono o tarjetas de crédito mediante expresiones regulares) para asegurar que esos datos confidenciales nunca salgan de nuestra infraestructura. También es donde frenamos ataques de Prompt Injection antes de gastar tokens procesándolos.
<br><br>
La segunda frontera crítica es el Output Guardrail (o Output Guard). Esta barrera se ejecuta después de que el LLM nos devuelve su respuesta, pero antes de que el usuario final la vea. Dado que no podemos controlar de forma determinista lo que genera el modelo, el Output Guard actúa como una red de seguridad para atrapar los fallos. Acá validamos si el modelo alucinó, si rompió el formato JSON que le pedimos, si empezó a usar lenguaje tóxico o si mencionó a nuestros competidores directos.
<br><br>
Bajo el capó, cualquiera de estos guardrails no es más que una función conocida como Validator (validador). Un validator recibe una cadena de texto (el input del usuario o el output del LLM) y ejecuta una regla de negocio. Esta regla puede ser tan simple como un if/else con un regex, o puede invocar modelos de Machine Learning locales y pequeños (como un clasificador Zero-Shot o un modelo de Natural Language Inference para detectar alucinaciones). El validator siempre debe devolver un resultado booleano (Pass o Fail).
<br><br>
¿Qué pasa cuando un validator devuelve un Fail? Acá es donde entra el diseño del flujo de control. Si un Input Guardrail falla (ej. el usuario intentó un hackeo), cortocircuitamos el proceso y devolvemos un mensaje de error estático inmediatamente, ahorrando latencia y costos de API. Si un Output Guardrail falla, la lógica de tu código puede ejecutar un retry (volver a llamar al LLM automáticamente, ya que al ser probabilístico podría acertar en el segundo intento) o activar un fallback (devolver una respuesta predefinida segura como "No puedo ayudarte con ese tema").
<br><br>
Aunque el flujo principal protege la entrada y la salida, en sistemas más complejos existe una tercera frontera: el Retrieval Guardrail. Este interceptor se ubica en el medio de una arquitectura RAG, justo después de consultar tu base de datos vectorial y antes de ensamblar el prompt final. Su trabajo es auditar la calidad y relevancia de los fragmentos (chunks) recuperados. Si la base de datos devuelve basura, el Retrieval Guardrail la descarta para evitar el problema de "basura entra, basura sale", asegurando que el LLM solo reciba contexto de altísima calidad.
<br><br>
Finalmente, como arquitecto siempre tenés que considerar el trade-off de latencia. Cada guardrail que inyectás en el flujo de datos suma milisegundos (o segundos) al TTFT (Time To First Token). Por eso es vital mantener este enfoque: escribir funciones de validación limpias, ligeras y ejecutar validadores pesados solo cuando el contexto del negocio lo exija estrictamente, evitando sobrecargar el sistema con frameworks de orquestación lentos e innecesarios.
