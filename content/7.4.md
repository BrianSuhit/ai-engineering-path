Hoy en día, modelos como Gemini ofrecen ventanas de contexto masivas que alcanzan hasta los 2 millones de tokens. Sin embargo, el hecho de que *podamos* inyectar gigas de texto o historiales interminables en una sola petición no significa que *debamos* hacerlo. A medida que agregamos miles de tokens, el modelo comienza a sufrir de **Context Rot** (pudrición del contexto). Llenar la memoria RAM del modelo con ruido provoca distracciones, confusión por información irrelevante y choques de datos contradictorios, lo que degrada drásticamente la capacidad de razonamiento del sistema. Además, procesar textos inmensos en cada turno dispara la latencia y los costos.
<br><br>
Para sobrevivir en esta economía de tokens y mantener el rendimiento, los arquitectos de IA aplican dos tácticas de infraestructura:
<br><br>
1. **Context Compaction (Compresión y Poda):** Consiste en editar activamente el contexto. En lugar de enviar un historial de chat infinito, podamos (*prune*) la información irrelevante. Si una herramienta devolvió 5,000 líneas de *logs* hace diez turnos y el modelo ya sacó su conclusión, eliminamos esa salida cruda del array de memoria. También podemos pedirle a un modelo más pequeño que resuma la conversación, conservando las decisiones arquitectónicas pero descartando la charla trivial.
2. **Context Caching:** Para el conocimiento que es pesado pero estático (como un manual técnico de 500 páginas, un repositorio de código entero o las instrucciones estrictas del sistema), usamos el almacenamiento en caché nativo del proveedor. En la nube de Google, esta técnica mantiene los documentos "congelados" en la memoria activa del servidor, permitiendo consultas ultrarrápidas y reduciendo los costos de procesamiento de esos tokens hasta en un 90%.
<br><br>
Acá tenés cómo se ve la arquitectura de **Context Compaction** en código utilizando la API de **Gemini**. Simulamos cómo un sistema en producción poda (*prunes*) un historial gigante manipulando directamente el array de **`steps`** antes de enviarlo, evitando saturar al modelo con basura vieja:
<br><br>

```python
from dotenv import load_dotenv
from google import genai

# Cargamos el entorno e inicializamos el cliente
load_dotenv()
client = genai.Client()

system_instruction = "You are a strict data analyst. Always provide concise answers."

# Simulamos un array masivo de 'steps' proveniente de una sesión muy larga.
# En producción, esto contendría decenas de turnos y salidas de herramientas (tool outputs) viejas.
massive_history_steps = [
    {"type": "user_input", "content": [{"type": "text", "text": "Analyze log file 1..."}]},
    {"type": "model_output", "content": [{"type": "text", "text": "Log 1 shows a Redis timeout."}]},
    # ... cientos de turnos y 5,000 líneas de logs viejos que ya no aportan valor ...
    {"type": "user_input", "content": [{"type": "text", "text": "Analyze log file 84..."}]},
    {"type": "model_output", "content": [{"type": "text", "text": "Log 84 shows high CPU usage."}]},
]

# CONTEXT COMPACTION (Pruning):
# En lugar de enviar todo, podamos el historial para retener estrictamente los últimos 2 turnos.
# Esto elimina el 'Context Distraction', limpia la memoria RAM del modelo y ahorra miles de tokens.
compressed_steps = massive_history_steps[-2:]

# Agregamos el nuevo turno (la consulta actual del usuario)
compressed_steps.append({
    "type": "user_input",
    "content": [{"type": "text", "text": "Based on the recent logs, what is the current system status?"}]
})

# Ejecutamos la inferencia con el contexto limpio y compacto.
# Nota: En la API de Interactions, si usáramos 'previous_interaction_id' en lugar de enviar
# el historial manual, el servidor aplicaría Context Caching de forma implícita para ahorrar costos.
interaction = client.interactions.create(
    model="gemini-3.5-flash",
    system_instruction=system_instruction,
    store=False, # Modo sin estado para inyectar nuestro contexto compactado manualmente
    input=compressed_steps
)

print(interaction.output_text)

```
