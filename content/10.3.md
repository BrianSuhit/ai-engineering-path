Cuando un usuario ingresa un *prompt* o pregunta, el sistema RAG inicia la fase de recuperación (*Retrieval*). Primero, la consulta del usuario se convierte en un vector utilizando el mismo modelo de *embeddings* que usamos para guardar los datos. Luego, la **Vector Database** ejecuta una búsqueda rápida (generalmente usando *Approximate Nearest Neighbors* o ANN) para encontrar y devolver los *K* fragmentos (*chunks*) matemáticamente más similares a la pregunta.
<br><br>
Sin embargo, acá surge un problema de arquitectura: **la similitud matemática rápida no siempre garantiza relevancia semántica real**. A veces, el sistema recupera fragmentos que comparten palabras clave con la pregunta, pero que no contienen la respuesta exacta. Si inyectamos estos fragmentos irrelevantes al LLM, la calidad de la respuesta final se desploma.
<br><br>
Para solucionar esto sin destruir la latencia, los *AI Engineers* implementamos un patrón de búsqueda en dos etapas introduciendo un **Reranker** (reordenador). Funciona así:
<br><br>
1. **Búsqueda inicial rápida:** La Vector DB recupera una lista "larga" de candidatos (ej. los 50 mejores resultados) en milisegundos.
2. **Reranking preciso:** Pasamos la consulta del usuario y esos 50 candidatos por un modelo de *Reranking* especializado (como *Cohere Rerank* o un modelo *Cross-Encoder* puro). Este modelo evalúa profundamente la consulta y cada documento *al mismo tiempo*, asignándoles una nueva puntuación de relevancia real.
3. **Filtrado final:** Tomamos solo los 3 o 5 mejores resultados post-reranking y esos son los que finalmente le pasaremos al LLM.
<br><br>
Con este diseño, obtenemos lo mejor de ambos mundos: la velocidad extrema de la base de datos vectorial para filtrar el ruido masivo, y la precisión quirúrgica del *Reranker* para elegir la información exacta.
