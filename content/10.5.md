Ya entendés la arquitectura y todos los pasos lógicos para armar un sistema RAG. Ahora el desafío de ingeniería es: ¿cómo conectamos la base de datos, el modelo de embeddings y el LLM en nuestro código? Para esto tenés dos grandes caminos: escribir el código de conexión vos mismo o usar un framework de orquestación.
<br><br>
**Using SDKs Directly (Código puro):** Acá implementás el flujo escribiendo la lógica manualmente, utilizando de forma directa las APIs de los modelos (como el SDK de OpenAI o Gemini) y de tu *Vector DB*, sin intermediarios.
- **Por qué usarlo:** Te da un control absoluto y transparencia total sobre el flujo de datos, evitando la complejidad o las abstracciones innecesarias que a veces meten los frameworks pesados. Es ideal para optimizar la latencia y facilitar la depuración.
<br><br>
**Orchestration Frameworks (Las herramientas del ecosistema):** Son librerías que ya resolvieron el "código pegamento" por vos y te dan bloques prefabricados. Las más usadas en la industria son:
- **LangChain:** El framework de orquestación más popular. Te proporciona abstracciones modulares para encadenar (*chain*) componentes como modelos, prompts, memoria y herramientas de recuperación de una forma estandarizada.
- **LlamaIndex:** Es un framework de datos diseñado específicamente para conectar fuentes de datos personalizadas con LLMs. Es la herramienta estrella para facilitar la ingestión, estructuración y recuperación de datos en arquitecturas RAG, incluyendo métricas para evaluar la fidelidad del contexto.
- **Haystack:** Un framework *end-to-end* súper robusto para construir *pipelines* de procesamiento de lenguaje natural y búsqueda semántica, con un enfoque muy claro en sistemas de preguntas y respuestas (QA).
- **RAGFlow:** Un motor open source enfocado específicamente en la comprensión profunda de documentos complejos, optimizando el flujo para que la IA genere respuestas precisas basadas en datos reales de una empresa.
<br><br>
Como *AI Engineer*, tu trabajo es elegir la herramienta correcta. Si necesitás un prototipo rápido con muchas integraciones, un framework es ideal. Si vas a producción y necesitás milisegundos de latencia sin "magia negra" en el código, ir directo con los SDKs suele ser la mejor decisión.
