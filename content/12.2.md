Para entender cómo viajan los datos en este ecosistema, hay que pensar en MCP como una arquitectura tradicional cliente-servidor, pero adaptada específicamente para la Inteligencia Artificial. El flujo de datos se divide en tres roles muy claros que separan la "toma de decisiones" de la "comunicación técnica".
<br><br>
El **MCP Host** es la aplicación final orientada al usuario donde el modelo de IA "vive" y opera, como por ejemplo la aplicación de escritorio de Claude, la interfaz de ChatGPT o un IDE moderno como Cursor. A nivel de arquitectura, este host es el gran orquestador: captura tu *prompt*, mantiene la memoria de la conversación, inicia las conexiones y toma la decisión lógica de qué herramienta necesita usar el modelo.
<br><br>
Adentro de este Host opera el **MCP Client**, que funciona estrictamente como un adaptador o mensajero de bajo nivel. Mientras que el modelo (el Host) decide qué acción quiere realizar, el Cliente MCP es la pieza de código que sabe cómo "hablar" en el protocolo MCP para negociar la conexión y transmitirle las instrucciones correctas al exterior.
<br><br>
Por último, del otro lado de la red tenemos al **MCP Server**, que es el servicio externo que realmente posee los datos o las capacidades de ejecución. Un servidor actúa como un "envoltorio" (*wrapper*) alrededor de tu infraestructura (como una base de datos corporativa o una API local), exponiendo sus funciones en un formato estandarizado para que cualquier cliente MCP del mundo pueda descubrirlas e invocarlas de forma segura.
