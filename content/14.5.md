En el diseño de sistemas con IA, la última frontera de protección es la puerta de salida. Incluso si tus prompts están perfectamente diseñados y tu arquitectura RAG recupera la información correcta, la naturaleza probabilística del LLM significa que siempre existe el riesgo de que el modelo se equivoque, alucine datos o rompa el formato estructural que le solicitaste.
<br><br>
Los Output Guardrails son interceptores lógicos que se ejecutan después de que la API del modelo te devuelve su respuesta, pero estrictamente antes de que ese texto sea procesado por tu base de datos o mostrado en la pantalla del usuario final. Su misión es actuar como una red de seguridad para atrapar los fallos en tiempo real, proteger la reputación de tu marca y garantizar que el sistema no colapse por recibir datos basura.
<br><br>
Para mantener la consistencia con nuestro caso de estudio, seguimos operando el chatbot de "Alfredo's Pizza Cafe". Sabiendo que el motor de inferencia puede fallar, vamos a implementar tres escudos de salida puramente lógicos:
<br><br>
**Competitor Blocker:** Sirve para proteger el negocio y la marca. ¿Por qué lo usamos? Porque si un usuario le pregunta al bot dónde comer la mejor porción de la ciudad, el modelo podría alucinar basándose en sus datos de pre-entrenamiento y recomendar a "Pizza by Alfredo" o "Dominos". Este interceptor escanea la respuesta generada y, si detecta a la competencia, descarta el texto y lo reemplaza por un mensaje prefabricado seguro.
<br><br>
**Format Validator (JSON/Structure):** Sirve para evitar que tu aplicación de software colapse. ¿Para qué lo usamos? Si le pedimos al modelo que extraiga el resumen de un pedido en formato JSON para enviarlo directamente al sistema de la cocina, y el modelo decide ser educado y agregar un "¡Claro, aquí tienes tu pedido!" antes de las llaves {}, tu código se va a romper. Este validador intenta parsear el texto matemáticamente y, si falla, frena la ejecución o activa un retry.
<br><br>
**Output PII Redaction:** Sirve para evitar fugas de datos de terceros. ¿Por qué lo usamos? En arquitecturas RAG avanzadas, el modelo podría leer un documento interno que dice "El cliente Juan (tel: 555-1111) pidió una grande de muzza" y escupirlo por error en el chat de otro usuario distinto. Este escudo detecta patrones sensibles en la salida y los enmascara antes de imprimirlos en pantalla.
<br><br>
A continuación, vamos a construir la arquitectura de estos validadores paso a paso, separando la lógica en funciones modulares.
<br><br>
**Paso 1: El escudo contra la competencia (Competitor Blocker)**
Esta función toma la respuesta ya generada por el LLM y la escanea contra una lista negra de reglas de negocio. Si falla la validación, cortocircuita el texto del modelo y devuelve un fallback (plan de contingencia).

```python
def block_competitors(llm_response: str) -> tuple[bool, str]:
    """
    Escanea la respuesta del modelo en busca de competidores prohibidos.
    Retorna (is_safe, final_message).
    """
    competitors = ["pizza by alfredo", "dominos", "pizza hut"]
    response_lower = llm_response.lower()
    
    for rival in competitors:
        if rival in response_lower:
            # El modelo falló la validación
            fallback = "I'm sorry, as an Alfredo's Pizza Cafe bot, I only focus on our delicious menu!"
            return False, fallback
            
    return True, llm_response
```
<br>

**Paso 2: El validador de formato (Format Validator)**
En lugar de usar expresiones regulares frágiles para validar estructuras complejas, usamos la librería nativa de Python para forzar el parseo del JSON. Si el modelo escupe texto libre o un JSON roto, atrapamos la excepción.

```python
import json

def validate_json_format(llm_response: str) -> tuple[bool, str]:
    """
    Verifica de forma estricta que la salida del LLM sea un JSON parseable.
    """
    try:
        # Intentamos cargar el texto como un diccionario real de Python
        parsed_data = json.loads(llm_response)
        return True, llm_response
    except json.JSONDecodeError:
        error_fallback = '{"error": "System failure: The model did not return a valid JSON."}'
        return False, error_fallback
```
<br>

**Paso 3: El Orquestador y el Flujo de Datos**
Acá es donde el flujo arquitectónico se invierte por completo respecto a los Input Guardrails. Sacamos las validaciones previas para enfocarnos únicamente en atrapar la salida.
El orquestador primero dispara la petición HTTP hacia la Interactions API de Gemini. Recién cuando la respuesta cruda vuelve a nuestro servidor, la hacemos pasar por la cascada de Output Guardrails antes de hacer el return final hacia el cliente.

```python
from google import genai

client = genai.Client()

def chat_with_alfredo_safe_output(user_input: str) -> str:
    
    # 1. Call the LLM API blindly (Input guards removed for this example)
    try:
        interaction = client.interactions.create(
            model="gemini-3.5-flash",
            system_instruction="You are a customer support bot for Alfredo's Pizza Cafe.",
            input=user_input
        )
        raw_llm_response = interaction.output_text
        
    except Exception as e:
        return f"API Connection Error: {e}"
        
    # 2. Run Competitor Blocker Guardrail on the raw generated text
    is_safe, final_response = block_competitors(raw_llm_response)
    
    if not is_safe:
        # Overwrite the hallucinated/unsafe response with our safe fallback
        return final_response
        
    # 3. If everything is perfect, return the actual model response to the user
    return final_response
```
<br>
Con este diseño arquitectónico, logramos contener el riesgo probabilístico del modelo. Si Gemini sufre una alucinación y decide recomendarle "Pizza by Alfredo" a un cliente, el usuario jamás llegará a leerlo. Nuestro código interceptará el texto anómalo en milisegundos y lo reemplazará por una salida segura, garantizando la fiabilidad del sistema en producción.
