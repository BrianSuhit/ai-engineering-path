A nivel de arquitectura, una vez que el cliente y el servidor MCP saben qué datos van a intercambiar, necesitan una ruta física para comunicarse. MCP utiliza **JSON-RPC** como su formato de mensaje estándar para estructurar las solicitudes, pero la verdadera flexibilidad del protocolo radica en su *Transport Layer*, que soporta dos métodos principales para mover estos mensajes: conexiones locales y conexiones remotas.
<br><br>
Para conexiones locales, el protocolo utiliza **Stdio (Standard Input/Output)**. En este escenario de cero latencia, el cliente MCP (por ejemplo, tu IDE Cursor o la app de escritorio de Claude) ejecuta el servidor directamente como un subproceso en la misma máquina (lanzando internamente un comando como **`python server.py`**). Esta arquitectura es el estándar de oro para el desarrollo y la privacidad, ya que garantiza que los datos sensibles o archivos locales nunca salgan hacia internet ni generen tráfico de red.
<br><br>
Por otro lado, para conectar sistemas distribuidos, MCP utiliza **SSE (Server-Sent Events) over HTTP**. Esto permite que un cliente se comunique con servidores alojados en la nube. Además, un patrón arquitectónico muy común acá es el *Tunneling*: si estás programando en tu máquina local pero necesitás conectar tu servidor MCP a un cliente remoto (como la interfaz web de ChatGPT), podés usar herramientas como **`ngrok`** o comandos nativos como **`mcp-use tunnel`** para exponer tu **`localhost`** de forma segura mediante una URL pública temporal.
